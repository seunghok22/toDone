## 1. 프레임워크 및 기술 스택 (Tech Stack)

- **데스크톱 프레임워크:** **Tauri** (앱의 핵심 뼈대 및 OS 환경 제어)
- **프론트엔드:** **React** (Vite 툴체인으로 빌드하여 번들 사이즈 최소화 및 빠른 개발 환경 구성)
- **로컬 데이터베이스:** **Tauri Plugin SQL (SQLite)** (백엔드 없이 사용자 PC 로컬에 DB 구축)
- **스타일링:** **Tailwind CSS**를 기본으로 사용하되, macOS 네이티브 UI 느낌을 가장 잘 살릴 수 있는 컴포넌트 프레임워크(예: **shadcn/ui**)를 탐색하여 적극 도입한다.

## 2. 아키텍처 및 데스크톱 앱 정책 (Architecture)

- **트레이 앱(Tray App) 구조:** Mac 상단 메뉴바(또는 Windows 작업 표시줄)의 아이콘을 클릭했을 때 창이 나타나고, 다른 곳을 클릭하면 숨겨지는(Hide/Blur) 프레임리스(Frameless) 윈도우 형태로 동작해야 한다.
- **리소스 최적화:** 창이 닫혀있는 백그라운드(Idle) 상태에서는 불필요한 프론트엔드 렌더링을 차단하고 리소스(CPU/메모리) 점유율을 0에 가깝게 최소화한다.
- **데이터 통신:** 외부 백엔드 서버를 두지 않고, React 프론트엔드에서 Tauri의 `invoke` 명령어를 호출하여 로컬 SQLite DB와 직접 데이터를 주고받는다.

## 3. UI/UX 및 스타일링 가이드 (Styling Guide)

- **디자인 에셋 연동:** 전체적인 화면 흐름과 레이아웃은 **Stitch**를 통해 생성한 디자인 데이터를 MCP로 불러와 최우선으로 반영한다.
- **macOS 네이티브 룩앤필 (필수 적용):**
    - OS 기본 시스템 폰트(`apple-system`, `BlinkMacSystemFont`, `SF Pro` 등)를 강제 적용한다.
    - 투명도와 Tailwind의 `backdrop-blur` 유틸리티를 조합하여 애플 특유의 프로스티드 글래스(Frosted Glass, 비치는 배경) 질감을 구현한다.
    - 웹사이트처럼 보이지 않도록 앱 전체 컨테이너에 텍스트 드래그 방지(`select-none`) 처리를 한다.

## 4. 빌드 및 자동화 정책 (CI/CD)

- 코드가 `main` 브랜치에 병합되거나 버전 태그가 생성될 때, GitHub Actions를 통해 Mac용 실행 파일과 Windows용 실행 파일(`.exe`)을 자동 빌드하여 **GitHub Releases**로 배포한다.

## 5. **폴더 구조 및 Atomic Design (아토믹 디자인) 패턴 적용**

AI가 컴포넌트를 무한정 크게 만드는 것을 막기 위해 명확한 폴더 분리 규칙을 제공합니다.

- **Atoms (`src/atoms`):** 버튼, 체크박스, 텍스트 입력창, 아이콘 등 더 이상 쪼갤 수 없는 최소 단위. (이때 `shadcn/ui`나 Tailwind로 스타일링을 끝냅니다.)
- **Molecules (`src/molecules`):** 원자들을 묶은 단위. (예: 할 일 항목 1개 = 체크박스 + 텍스트 + 삭제 버튼)
- **Organisms (`src/organisms`):** 분자들을 묶어 만든 독립적인 영역. (예: 주간 작업 리스트, 캘린더 위젯)
- **Templates / Pages (`src/pages`):** 유기체들을 모아둔 뼈대와 최종 트레이 팝업 화면.

## **6. 코드 컨벤션 (Code Conventions)**

- **명명 규칙 (Naming):** 컴포넌트와 파일명은 `PascalCase`(예: `TaskItem.tsx`), 함수와 변수는 `camelCase`, 상수는 `UPPER_SNAKE_CASE`를 강제합니다.
- **조기 반환 (Early Return):** 조건문이 깊어지는(Depth가 깊어지는) 것을 방지하기 위해, 예외 상황은 함수 도입부에서 즉시 `return` 처리하도록 지시합니다.
- **상태 관리 제한:** 불필요한 전역 상태를 남발하지 않고, 부모-자식 간의 Props 전달을 우선하되 깊이가 3단계를 넘어가거나 많은 곳에서 쓰이는 경우는 Zustand를 사용합니다.

## **7. Git 커밋 및 테스트 검증 정책 (Agent Workflow Rules)**
AI 에이전트는 코드를 작성하고 저장소에 반영(Commit)할 때, 반드시 아래의 프로세스와 규칙을 스스로 준수해야 한다.

[1단계: Pre-commit 자동 검증]

- 코드를 수정한 후 커밋하기 전에, 반드시 내장된 터미널을 이용해 npm run lint (또는 타입스크립트 에러 체크)와 npm run build를 실행하여 컴파일 에러가 없는지 스스로 확인한다.
- 핵심 로직(상태 관리, 날짜 계산 등)을 추가했을 경우 단위 테스트(Vitest 등)를 작성하고 통과하는지 확인한다.
- 에러가 발생하면 커밋을 중단하고, 스스로 에러 원인을 분석하여 코드를 수정한 뒤 다시 테스트를 통과할 때까지 반복(Loop)한다.

[2단계: 커밋 메시지 컨벤션 (Conventional Commits)]

에러 없이 검증이 끝난 코드만 Git에 커밋하며, 반드시 아래의 접두사(Prefix) 포맷을 준수하여 한글로 상세히 작성한다.

- feat: 새로운 기능 추가 (예: feat: 주간 캘린더 UI 및 Zustand 상태 연동)
- fix: 버그 수정
- refactor: 코드 리팩토링 (기능 변화 없이 구조만 개선)
- style: UI/UX 컴포넌트 스타일링, Tailwind 클래스 변경
- chore: 패키지 설치, 빌드 설정, 기타 잡무
- docs: 문서화 작업 (README.md, 요구사항 문서 등)

[3단계: 작업 단위 분할]

- 한 번의 커밋에 너무 많은 변경 사항을 뭉뚱그려 담지 않고, 하나의 원자(Atom) 컴포넌트나 단일 기능이 완성될 때마다 잘게 쪼개서 자주 커밋(Frequent Commit)한다.

## 8. 작업 일지 및 문서화 자동화 (Changelog & Documentation)

AI 에이전트는 본인의 작업 내역을 사용자가 쉽게 파악할 수 있도록 지속적으로 문서를 최신화해야 한다.

- **CHANGELOG 자동 기록:** 유의미한 기능 구현, UI 변경, 또는 버그 수정을 완료하고 커밋을 진행할 때마다 프로젝트 루트에 있는  `docs/history.md` 파일에 작업 내역을 누적해서 기록한다.
- **기록 포맷:** 작성 시 다음 양식을 따른다.
    - `## [날짜/시간] 작업 요약` (예: ## 2026-03-23 캘린더 UI 뼈대 구현)
    - `### ✨ Added (추가된 기능)`
    - `### ✨ commits (관련 커밋 리스트)`
    - `### 🐛 Fixed (해결된 문제 및 버그)`
    - `### 📝 Next Steps (다음에 진행할 추천 작업)`
- **설계 문서 동기화:** 작업 중 부득이하게 기술 스택을 변경하거나 새로운 상태 관리 구조를 도입했다면, `01_requirements.md`나 `02_conventions.md` 문서도 함께 수정하여 **코드와 기획 문서가 항상 100% 일치**하도록 유지한다.
